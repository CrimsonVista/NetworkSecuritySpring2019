Post-Interstellar Mining Protocol Layer Security (PLS) Request For Comments

Abstract

Post-Interstellar Mining Protocol Layer Security (PLS) is a proposed transport layer security protocol that provides key authentication and confidentiality on the Playground environment. PLS is a simplified, TLS-like protocol. PLS simplifies the protocol by using a simpler ciphersuite for encryption.

Status of this Memo

This document specifies a Playgrounds standards track protocol for the Playground Experiment, and requests discussion and suggestions for improvements. Distribution of this memo is unlimited.

2.  Motivation

For applications running in the Playground environment to reliably communicate, there should be a secure way for two entities to send and receive data without fear of a third-party intercepting their communication. The PLS protocol will have the following requirements met:

  1. Confidentiality: Any data sent using PLS will only be visible to the sender and receiver.
  2. Integrity: If data is modified using PLS, it will be detected.
  3. Authentication: Both the client and the server will be authenticated using certificates.

This protocol is closely related to the TLS 1.3. Specifically, our protocol follows the same handshake agreement with the addition of the client verification.

The handshake is where the client and server will negotiate protocol versions, which algorithms will be used, and what the shared secret key will be moving forward.

There are two main phases of the PLS handshake: key transport and authentication. For key transport, the client will send its hello and certificate. The server will send their hello message and their certificate. For authentication, the client and server verify the certificates with the appropriate certificate authority and generate keys with the premaster secret. After this, a change cipher spec message will be sent to the server to let the server know to use the generated keys for bulk transfer. Then the client will send their finished message, and the server will send their finished message after confirming the cipher spec change.

3.1.  Handshake Description

Client Hello:

When the client wants to connect to a specific server, it will send a hello message. This hello message is a random message. Along with the hello, the client will send the ciphersuites it supports and a certificate with their public key.

Server Authenticates Client Certificate:

The server will use a trusted certificate authority (CA) to verify the client's certificate.

Server Hello:

The server will respond with a hello message, which is also random. With the hello message, the server will send their certificate.

Client Authenticates Server Certificate:

When the client receives the server's certificate, it will verify it using a certificate authority (CA). The CA is a trusted source that knows which certificates are valid and which are not.

Premaster Secret:

The premaster secret will be sent to the server, encrypted under the server's public key. The server will send it back encrypted under the client's public key. This premaster secret will establish trust between the client and server, and it will be used for the premaster secret that is sent afterwards.

Client Finished:

The client will send a finished message to let the server know that they are done with verification. The client will also scipher spec change request to let the server know to switch to symmetric encryption.

Server Finished:

The server will respond with a finished message along with the client's cipher spec change. This will signal to the client that the server is done verifying and that the server knows to use the derived shared keys.

3.2.  Packet Description

        from playground.network.packet import PacketType
        from playground.network.packet.fieldtypes
                              import UINT32, STRING, BUFFER, BOOLEAN
        class PLSPacket(PacketType):
            DEFINITION_IDENTIFIER = "PLS.Packet"
            DEFINITION_VERSION = "1.0"
            FIELDS = [
              ("Type", STRING),
              ("Premaster_Secret", UINT32) {OPTIONAL}
              ("Random", UINT32){OPTIONAL},
              ("Certificate", LIST(BUFFER)){OPTIONAL},
              ("Encrypted_Data", BUFFER)
            ]

As shown in the figure above, the PLS packet will have the following fields:

Type:

The type can be one of five options: Hello, KeyTransport, Finished, Data, and Close. Hello is the first type that will initiate the PLS handshake. Finished is used by each side to let the other know that their side is finished with negotiating the protocol, ciphersuite, shared key, and with verifying the certificates. Data is used to let each side know that there is data coming through. Close will alert the other side to stop sending any data.

Random:

The random is a 32 bit randomly generated number that is sent during the hello message.

Premaster Secret:

The premaster secret is a 32 bit randomly generated by the client. This key will be used in creating the master secret. It is initially encrypted using the server's public key and sent to the server. When the server sends the premaster secret back, they will encrypt it using the client's public key.

Certificate:

The certificate field is for each side to send their signed certificate chain. which is made up of the root certificate, the intermediary certificate, and the individual certificate. This field is optional after the handshake.

Encrypted Data:

This field is the encrypted data that we want to send between the client and the server.

4.  Certificate

Certificates provide information as to whether the client and server are who they are claiming to be. For PLS, we will use the X.509 certificate as shown below: 



-+-+-+-+-+-+-+-+-+-+-+-+-
| PLS.509 Certificate   |
+-+-+-+-+-+-+-+-+-+-+-+-+
|     Version Number    |
+     Serial Number     +
|     Validity Period   |
+       Issuer Name     +
|      Subject Name     |
+       Public Key      +
|  Public Key Algorithm |
+ Certificate Signature +
|    Issuer Unique ID   |
+   Subject Unique ID   +
|+-+-+-+-+-+-+-+-+-+-+-+|

For PLS, there will be certificates for each individual, that will lead to an intermediary CA (ICA). The individual certificates are self signed, meaning that the issuer and subject name will be the same. The intermediary CA will lead to the Root CA (the bank owner). This is called a certificate chain. There will be one Intermediary CA for each address block. For example, there will be Intermediary CA's for 20191.1, 20191.2, 20191.3, 20191.4, 20191.5, and 20191.6. On the Intermediary certificate, the issuer name is the ICA and the subject name will be the individual's name. ACertificates can be created using openssl. Certificates will be transported in in a serialized PEM format.

5.  Key Derivation

The master key is a concatenation of the client's hello random, the server's hello random, and the premaster secret. It is encrypted using RSA of key size 2048 and OAEP padding with the server's public key. Both sides will take that key and use HKDF with SHA256 as the hashing function to derive the IV, read and write keys. Since this is symmetric encryption, the read key for the client will be the write key for the server, and the write key for the client will be the read key for the server.

6.  Algorithms

For PLS, there will only be one supported cipher suite: PLS_RSA_AES_256_GCM_SHA256.

1. PLS is the protocol.

2. RSA is the encryption algorithm for key transport and authentication.

3. For bulk data transfer, the encryption algorithm PLS uses is AES 256 GCM. ****Please see note below.****

4. SHA_256 will be the hashing algorithm used for the key derivation function.

****NOTE: For AES 256 GCM - We will use Authenticated Encryption with Associated Data (AEAD). This provides integrity for the data. We will use the default 16-bit tag that is appended at the end of the data.

7.  Error Handling

PLS will assume that a connection is made over a reliable transport. Otherwise, it will drop packets that have the following errors:

1. The subject name on the certificate does not match the name of who sent it.

2. The signature on the certificate cannot be verified.

3. The Intermediary CA cannot verify the certificate.

4. The Root CA cannot verify the certificate.

If the error is not corrected after the second attempt, a close packet will be sent, and a connection will not be made.

8.  PLS Shutdown

Either side has the option to close the connection. If one side send the other a packet of type 'Close,' then no more data will be sent on the channel.

9.  Security Considerations

There are only two security considerations:

1. Only one ciphersuite is used.

2. PLS is the only protocol that the server will support.

3. Keys will not be reused during a session.

10.  PRFC Revision Process

Once a PRFC is officially published by the PETF, it cannot be modified further for any reason. A new PRFC must be submitted that identifies the errors and proposes to "update" or "supercede" the previous document.

A PRFC MUST use the "update" reference if the changes do not result in an incompatible protocol and the old protocol is not shown to be insecure. Changes to PRFC's that are "updates" include:

  1. Corrections to spelling, grammar, and other wording in the document
  2. Additional details about operations and operational considerations
  3. Additional parameters or interactions to the existing protocol

A PRFC MUST use the "supercede" reference if the changes result in an incompatible protocol OR if the previous protocol is demonstrated to be insecure.

Approval from the PETF MUST be obtained if approval was required for the original PRFC.

A PRFC submitted without PETF approval MUST be revised only by the same PLAYGROUND group that submitted the original.
